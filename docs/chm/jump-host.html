Jump Host / ProxyJump

Jump Host support allows the plugin to reach the target SSH server through another SSH server (bastion host) without using external ssh.exe.

How it works

  1. TCP connect to the jump host
  2. SSH handshake and authentication on the jump host
  3. Open a direct TCP tunnel to the target host using SSH direct-tcpip
  4. Run the normal target SSH session over that tunnel
Use case: Servers reachable only from a bastion/jump machine, common in segmented internal networks and production environments.

When to use Jump Host

  • Use it only with SSH (SFTP/SCP)
  • Do not use it with PHP Agent
  • Do not use it with PHP Shell
  • Do not use it with LAN Pair

Main dialog behavior

  • Use jump host and Jump... are enabled only when Transfer = SSH (SFTP/SCP)
  • For PHP and LAN Pair modes these controls are intentionally disabled
  • Jump-host settings remain stored in the profile even when the controls are currently disabled

Jump Host settings

  • Host - bastion hostname or IP
  • Port - usually 22
  • User - jump-host account
  • Password - jump-host password, if used
  • Public/Private key - separate key material for the jump host
  • Use SSH agent (Pageant) - optional jump-host authentication via Pageant

Two independent authentication layers

Jump Host and target server are separate SSH identities:

  • the jump host can use one user/key/password
  • the target host can use another user/key/password
  • each side has its own fingerprint verification

Fingerprint handling

  • Target host fingerprint is handled as usual
  • Jump host fingerprint is stored separately as jumpfingerprint
  • On first use or fingerprint change, the plugin asks for confirmation

INI keys

Typical profile keys:

[MyServer]
usejumphost=1
jumphost=bastion.example.com
jumpport=22
jumpuser=ops-user
jumppubkeyfile=C:\Keys\bastion.pub
jumpprivkeyfile=C:\Keys\bastion.ppk
jumpuseagent=1
jumpfingerprint=AA BB CC DD ...

Requirements and limits

  • The jump host must allow direct TCP forwarding / direct-tcpip channels
  • A normal non-root user is sufficient if the SSH server allows this feature
  • Classic HTTP/SOCKS proxies and Jump Host are different layers; do not treat them as the same thing

Failure cases

  • Jump host reachable, target unreachable from bastion
  • Authentication works on bastion, but direct-tcpip is disabled by server policy
  • Saved jump-host fingerprint changed
  • Wrong jump-host credentials with correct target credentials (or the reverse)
