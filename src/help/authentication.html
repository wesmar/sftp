Authentication Methods

The plugin supports multiple SSH authentication methods and tries them in the order best suited to your configuration.

Available Authentication Methods

Password Authentication

  • Standard username/password login
  • Password stored encrypted via Windows DPAPI (user-account scope)
  • Most widely supported method

Public Key Authentication

  • Uses an SSH key pair (public/private)
  • More secure than password — server never sees the private key
  • Supports PEM (OpenSSH) and PPK (PuTTY) formats
  • Encrypted keys (with passphrase) are supported

SSH Agent (Pageant)

  • Uses keys loaded in the running Pageant agent
  • No need to specify key files in the session
  • Falls back to other methods if no matching key is accepted
  • If Pageant is not running but pageant.lnk exists in the plugin directory, the plugin launches it automatically and retries

Keyboard-Interactive

Keyboard-interactive is a server-driven challenge-response mechanism. The server sends one or more prompts; the plugin answers each one.

  • Used by many hosting providers and corporate SSH servers
  • Common scenarios: standard password prompt, OTP / TOTP code, SMS verification code, security question
  • The plugin handles multiple prompts in sequence automatically — each prompt is shown to you in turn
  • If the server sends a single prompt that looks like a password prompt and you have a password saved in the session, the saved password is used automatically (no dialog shown)
  • If the server sends prompts the plugin cannot satisfy automatically, a dialog appears for each
2FA via keyboard-interactive: If your server requires a password + TOTP code (e.g. Google Authenticator), the plugin will first try the saved password, then prompt you for the OTP code. No special configuration is needed — just answer the prompts as they appear.

Authentication order (fallback chain)

  1. Pageant (if "Use SSH agent" is checked and Pageant is running)
  2. Key file (if a private key file is specified)
  3. Keyboard-interactive
  4. Password

Each method is tried only if advertised by the server. The chain stops at the first success.

Key File Formats

PEM Format (OpenSSH)

  • Standard OpenSSH private key format
  • Generated by ssh-keygen
  • File extension: .pem or no extension
  • RSA, ECDSA (nistp256/384/521), and Ed25519 supported

PPK Format (PuTTY) — native, no external tool needed

  • PuTTY native key format, generated by PuTTYgen
  • File extension: .ppk
  • The plugin converts PPK internally — puttygen.exe is not required and argon2.dll is not needed
  • Argon2 is statically compiled into the plugin

Key format detection

The plugin automatically detects whether a key is PEM or PPK from the file header. You do not need to tell the plugin which format is used.

Using Public Key Authentication

Setup steps

  1. Generate a key pair with ssh-keygen or PuTTYgen.
  2. Copy the public key to the server: append it to ~/.ssh/authorized_keys.
  3. In the plugin dialog, click the >> button next to Private key file and browse to your key.
  4. For PEM keys: the public key field can be left empty (derived automatically).
    For PPK keys: the public key field is also not required — the public key is embedded in the PPK file.
  5. If the key has a passphrase, you will be prompted when connecting.

Passphrase handling

  • Passphrase is prompted at connection time
  • It is not stored in the session (by design)
  • To avoid the prompt, load the key into Pageant — Pageant caches the unlocked key for the session

Using Pageant (SSH Agent)

Setup

  1. Start Pageant (included with PuTTY).
  2. Load your private key(s) into Pageant.
  3. In the plugin dialog, check Use SSH agent (Pageant).
  4. Connect — the plugin queries Pageant for available keys and tries each one.

Auto-launch

If Pageant is not running when you connect, the plugin checks for a file named pageant.lnk in the plugin directory (...\plugins\wfx\SFTPplug\). If found, it launches Pageant automatically, waits briefly, then retries agent authentication.

Common Issues

Key authentication fails

  • Verify the public key is in ~/.ssh/authorized_keys on the server
  • Check server-side permissions: chmod 700 ~/.ssh and chmod 600 ~/.ssh/authorized_keys
  • Enable detailed logging to see which key the plugin attempted

PPK conversion fails

  • Verify the PPK file is not corrupted — try opening it in PuTTYgen as a test
  • argon2.dll is NOT required — Argon2 is built into the plugin. If you see a DLL error, it is unrelated to PPK support
  • Make sure the PPK file path in the session dialog is correct and the file is readable

Pageant not detected

  • Check that Pageant is running (system tray icon)
  • Verify the key is loaded in Pageant (right-click tray icon → View Keys)
  • Ensure Use SSH agent (Pageant) is checked in the plugin dialog
  • If you want Pageant to start automatically, place a pageant.lnk shortcut in the plugin directory

Keyboard-interactive prompts

  • Answer each prompt as shown — the plugin passes your responses to the server in order
  • For 2FA, enter the OTP/TOTP code exactly as shown by your authenticator app
  • If you see unexpected prompts (e.g., a Linux "Last login" message), check whether your server sends informational messages during auth — these are displayed but do not require a response

Security Recommendations

  • Use key authentication when possible
  • Protect private keys with a passphrase and use Pageant to cache the unlocked key
  • Do not store passphrases in session profiles
  • Rotate keys periodically for sensitive systems
  • Keep the plugin directory access-controlled — Pageant auto-launch reads pageant.lnk from there
