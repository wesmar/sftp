Character Encoding and Codepages

The plugin handles character encoding for filenames and file contents. Proper encoding settings ensure correct display of non-ASCII characters.

Filename Encoding

UTF-8 Detection

The plugin can auto-detect server filename encoding:

  • Auto-detect: Plugin tries UTF-8 first, falls back to ANSI if needed
  • Force UTF-8: Always interpret filenames as UTF-8
  • Force ANSI: Use system codepage (e.g., Windows-1250, Windows-1251)

Supported Codepages

For ANSI mode, the following codepages are supported:

  • Windows: 1250 (Central European), 1251 (Cyrillic), 1252 (Western), 1253 (Greek), 1254 (Turkish), 1255 (Hebrew), 1256 (Arabic), 1257 (Baltic), 1258 (Vietnamese)
  • Asian: 932 (Japanese), 936 (Simplified Chinese), 949 (Korean), 950 (Traditional Chinese)
  • Other: 437 (DOS US), 850 (DOS Multilingual), 874 (Thai), 20866 (KOI8-R)

Transfer Mode

Binary Mode

Raw byte transfer without modification. Use for:

  • Executable files
  • Archives (ZIP, RAR, 7z)
  • Images, audio, video
  • Most modern text files (UTF-8, UTF-16)

ASCII Mode

Converts line endings during transfer:

  • Upload: LF → CRLF (Unix → Windows)
  • Download: CRLF → LF (Windows → Unix)
  • Use for: Plain text files, scripts, configs

Auto Mode

Uses wildcard patterns to determine transfer mode automatically.

Text File Type Patterns

Configure which files use ASCII mode:

  • Default patterns: *.txt;*.log;*.ini;*.cfg;*.xml;*.json
  • Multiple patterns separated by semicolons
  • Wildcards supported: * (any), ? (single char)
  • Case-insensitive matching

Pattern Examples

*.txt;*.log;*.ini          - Common text files
*.c;*.h;*.cpp;*.hpp        - Source code files
*.md;*.rst;*.adoc          - Documentation files
*.htm;*.html;*.css;*.js    - Web files

Line Ending Detection

The plugin can auto-detect line ending format:

  • LF (Unix): 0x0A - Linux, macOS, modern Windows tools
  • CRLF (Windows): 0x0D 0x0A - Traditional Windows
  • CR (Classic Mac): 0x0D - Legacy Mac (pre-OS X)

UTF-8 BOM Handling

The plugin preserves UTF-8 BOM (Byte Order Mark) during transfers:

  • BOM detected at file start
  • Preserved during binary transfer
  • Handled correctly in ASCII mode

Unicode in Paths

Full Unicode support for:

  • Local paths (Windows UTF-16)
  • Remote paths (UTF-8 or configured codepage)
  • Session names (stored in INI as UTF-8)

Common Issues

Garbled filenames after transfer

  • Check UTF-8 setting matches server configuration
  • Try forcing UTF-8 or forcing ANSI
  • Verify correct codepage for ANSI mode

Text files have wrong line endings

  • Ensure file matches text pattern for ASCII mode
  • Check if file is actually UTF-16 (needs binary)
  • Use binary mode for mixed-encoding files

Asian characters display incorrectly

  • Use UTF-8 mode if server supports it
  • Select correct Asian codepage (932/936/949/950)
  • Ensure system has appropriate fonts installed
Best practice: Modern servers typically use UTF-8. Start with UTF-8 auto-detect, then adjust if you see encoding issues.
